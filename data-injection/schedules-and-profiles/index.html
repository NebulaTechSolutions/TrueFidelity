    Schedules & Profiles¶

    Schedules define when and how injection messages are sent. Profiles bundle schedules and messages so you can reuse scenarios. This guide dives into each schedule type and offers practical examples.

    Schedule Types¶

    Once¶

    • Sends a message a single time after an optional delay.
    • Useful for triggering events or sending diagnostic requests.
    • Configure delay in seconds or milliseconds.

    Interval¶

    • Repeats a message at a fixed rate (e.g., every 100 ms).
    • Supports fractional intervals (0.1 s) and frequency notation (10 Hz).
    • Optionally align to playback time by enabling Sync with timeline.

    Burst¶

    • Sends a burst of N frames quickly (burst size) followed by a pause (cooldown).
    • Repeat bursts until stopped or after a defined duration.
    • Example: 50 frames burst, 5-second pause, repeat 10 times.

    Ramp¶

    • Gradually changes the interval between frames.
    • Specify start rate, end rate, duration, and ramp shape (linear, exponential).
    • Great for stress testing controllers by ramping from 2 Hz to 50 Hz.

    Custom¶

    • Import custom schedule scripts (if your edition supports it) using JSON or Python-based instructions.
    • Scripts allow conditional logic, randomization, and complex scenarios.

    Setting Up Schedules¶

    1. Select a message template in the injection panel.
    2. Choose the schedule type from the dropdown.
    3. Configure parameters (delay, rate, duration, repeat count).
    4. Preview the timeline graph to validate timing.
    5. Save your settings before starting injection.

    Tip

    Combine multiple messages with different schedules in the same profile to simulate realistic bus activity (e.g., heartbeat + sporadic fault message).

    Profiles¶

    Profiles collect messages and schedules into reusable packages.

    • Create profile: Configure messages and schedules, then click Save Profile.
    • Load profile: Click Load Profile, select a .tfinject file, and review the summary before starting.
    • Export profile: Share with teammates to reproduce tests.
    • Versioning: Include version numbers or dates in profile names (e.g., ABS_failover_v3.tfinject).

    Profiles store: - Channel selection - Message templates (ID, DLC, payload) - Schedule settings - Default duration and stop conditions

    Example Profiles¶

    Heartbeat Simulation¶

    • Message ID: 0x120
    • Payload: 01 7F 00 00 00 00 00 00
    • Schedule: Interval 100 ms
    • Use case: Simulate a sensor heartbeat during bench testing.

    Fault Injection Burst¶

    • Message ID: 0x3E0
    • Payload: FF FF FF FF 00 00 00 00
    • Schedule: Burst (burst size 30, cooldown 3 s, repeat 5 times)
    • Use case: Stress ECU error handling routine.

    Ramp Stress Test¶

    • Message ID: 0x245
    • Payload: AA 00 AA 00 55 55 00 01
    • Schedule: Ramp from 5 Hz to 50 Hz over 60 seconds, hold for 30 seconds, ramp down.
    • Use case: Validate firmware response to increasing data rates.

    Managing Profiles¶

    • Store profiles in a dedicated folder within the project (e.g., profiles/).
    • Document the purpose, message IDs, and expected outcomes in a README.
    • Use version control to track profile changes alongside firmware builds.

    Monitoring During Scheduled Injection¶

    • Watch the counters (sent, failed, rate) to ensure the schedule runs as expected.
    • Use annotations to mark the start and end of injection runs.
    • Verify actual timing using the Messages panel by filtering for injected IDs and checking timestamp spacing.

    Troubleshooting Schedules¶

    Issue Resolution
    Interval drift Check system load; high CPU usage can delay timers. Reduce other workload or use a dedicated machine.
    Burst stops early Ensure the repeat count is set correctly and there are no hardware errors.
    Ramp doesn't reach target rate Confirm the duration is long enough and the channel is not saturating.
    Profile fails to load The file may reference a channel or message format not available in this environment. Edit the profile or update hardware configuration.

    Best Practices¶

    • Test schedules with playback paused before mixing with live data.
    • Label injection profiles with project IDs and expected outcomes.
    • Keep profiles under version control with change logs.
    • Use separate profiles for destructive and non-destructive tests; never run destructive profiles on live systems without authorization.
