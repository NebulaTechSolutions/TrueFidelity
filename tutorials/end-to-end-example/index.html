    End-to-End Example: CAN Log Analysis Workflow¶

    This comprehensive tutorial demonstrates a complete TrueFidelity workflow for analyzing CAN bus data. You'll learn how to create a system, configure networks, deploy ECUs, capture live data, analyze logs with playback, and perform signal injection testing. This tutorial represents a realistic debugging scenario that automotive engineers face daily.

    Time Estimate¶

    • Initial Setup: 10-15 minutes
    • System Configuration: 15-20 minutes
    • Data Analysis: 20-30 minutes
    • Total Time: 45-65 minutes

    Prerequisites¶

    Before starting this tutorial, ensure you have:

    • TrueFidelity Desktop application installed and licensed
    • Docker Desktop running (or Mock Mode enabled for simulation)
    • Sample BLF log files (available in the examples folder)
    • A DBC file with signal definitions for your CAN network
    • Basic familiarity with CAN bus concepts and signal analysis

    Scenario Overview¶

    You're investigating intermittent issues in a vehicle's powertrain system. The symptoms include:

    • Unexpected RPM fluctuations during steady-state cruise
    • Occasional throttle response delays
    • Diagnostic trouble codes that appear and clear randomly

    Your task is to:

    1. Create a test bench system that mimics the vehicle network
    2. Replay captured log data to reproduce the issue
    3. Analyze signal patterns to identify root causes
    4. Test potential fixes using signal injection
    5. Document findings for the engineering team

    Part 1: System Setup and Configuration¶

    Step 1: Create a New System¶

    Start by creating a fresh system for your analysis work.

    1. Launch TrueFidelity Desktop:
    2. If you see an existing system, save any changes first

    3. The application starts in System Design mode by default

    4. Create the new system:

    5. Click File → New System from the menu bar

    6. Or use the keyboard shortcut Ctrl+N (Cmd+N on Mac)

    7. Configure system properties in the dialog:

    8. System Name: Enter PowertrainAnalysis
    9. Description: "Investigation of RPM fluctuations and throttle response"
    10. Author: Your name
    11. Version: Leave as 1.0.0
    12. Click Create

    The System Canvas opens with an empty diagram, ready for you to build the network topology.

    Step 2: Add the CAN Network¶

    You'll need a CAN bus to connect your ECUs and carry the vehicle signals.

    1. Open the Toolbox panel (left sidebar):
    2. If not visible, use View → Panels → Toolbox
    3. The Toolbox contains draggable components

    1. Add a CAN bus to the canvas:
    2. In the Toolbox, expand the Networks section
    3. Find CAN Bus in the list
    4. Drag it onto the center of the canvas
    5. The bus appears as a horizontal line with connection points

    1. Configure the CAN network:
    2. Click on the CAN bus to select it
    3. The Properties panel (right side) shows network settings
    4. Set these properties:
      • Name: Powertrain-CAN
      • Baud Rate: Will auto-populate from DBC
      • DBC File: Click Browse and select your powertrain.dbc file
      • Click Apply Changes at the bottom of Properties

    Step 3: Add ECUs to the System¶

    Now add the ECUs that will participate in the network communication.

    1. Add the Engine ECU:
    2. In Toolbox under Components, find Generic ECU
    3. Drag it above the CAN bus on the canvas

    4. Position it centrally above the bus

    5. Configure the Engine ECU:

    6. Click the ECU to select it
    7. In Properties, set:
      • Name: Engine-ECU
      • Type: Select Linux Generic x86_64
      • Container Base: ubuntu:22.04 (or your preferred base)
    8. Click Apply Changes

    1. Add the Transmission ECU:
    2. Drag another Generic ECU from Toolbox
    3. Place it to the right of the Engine ECU

    4. Configure in Properties:

      • Name: Transmission-ECU
      • Type: Linux Generic x86_64
      • Click Apply Changes

    5. Add the Gateway ECU:

    6. Drag a third Generic ECU from Toolbox
    7. Place it to the left of the Engine ECU
    8. Configure:
      • Name: Gateway-ECU
      • Type: Linux Generic x86_64
      • Click Apply Changes

    Step 4: Connect ECUs to the Network¶

    Each ECU needs to be connected to the CAN bus to communicate.

    1. Connect the Engine ECU:
    2. Hover over the Engine ECU box
    3. Small blue connection handles appear on the edges
    4. Click and drag from the bottom handle
    5. Drag the connection line down to the CAN bus
    6. Release when the bus highlights green (valid connection)
    7. A connection line appears linking them

    1. Connect the Transmission ECU:
    2. Repeat the same process:
    3. Hover over Transmission ECU
    4. Drag from bottom handle to CAN bus

    5. Release when highlighted

    6. Connect the Gateway ECU:

    7. Connect it the same way to complete the network
    8. All three ECUs should now be connected to the CAN bus

    1. Verify connections in Project Explorer:
    2. Look at the Project Explorer (left panel)
    3. Expand Networks → Powertrain-CAN
    4. You should see all three ECUs listed as connected

    Step 5: Save the System¶

    Before proceeding, save your work to prevent data loss.

    1. Save the system:
    2. Click File → Save or press Ctrl+S
    3. Choose a location for your .tfsys file
    4. Name it PowertrainAnalysis.tfsys

    5. Click Save

    6. Verify save status:

    7. The title bar should show "PowertrainAnalysis" without an asterisk
    8. The status bar (bottom) confirms "System saved"

    Part 2: System Deployment and Monitoring¶

    Step 6: Build and Deploy the System¶

    Transform your design into running containers.

    1. Start the build process:
    2. Click the Build System button in the toolbar
    3. Or use System → Build from the menu

    4. The build console appears at the bottom

    5. Monitor build progress:

    6. Watch for "Building ECU firmware..." messages
    7. Each ECU shows "Firmware ready" when complete
    8. Look for "Build completed successfully"
    9. If errors occur, check:
      • Docker Desktop is running
      • Network connectivity is available
      • Sufficient disk space exists

    1. Troubleshooting build failures:

    Docker not running: Error: Cannot connect to Docker daemon Solution: Start Docker Desktop and retry

    Network issues: Error: Failed to pull base image Solution: Check internet connection, retry

    Disk space: Error: No space left on device Solution: Clear Docker cache with: docker system prune

    Step 7: Start the System¶

    Deploy and run your ECU containers.

    1. Start the system:
    2. Click the Start button in the toolbar (green play icon)
    3. Or use System → Start from the menu

    4. Status bar shows "Starting system..."

    5. Monitor startup:

    6. The ECU Console opens automatically
    7. Watch for startup messages from each ECU
    8. Look for "ECU initialized" messages

    9. Typical startup takes 10-30 seconds

    10. Verify all ECUs are running:

    11. Check the status bar: "3 of 3 containers running"
    12. In Mock Mode, you'll see simulated container status

    Step 8: Switch to Monitoring Mode¶

    Monitor the health and status of your running system.

    1. Switch to Monitoring mode:
    2. Click View → Monitoring from the menu
    3. Or click the Monitoring button if available in toolbar

    4. The layout changes to show monitoring panels

    5. Review the monitoring panels:

    6. ECU Hardware Status (top) - Shows all ECUs and their states
    7. ECU Resource Monitor (middle left) - CPU and memory graphs
    8. ECU Task Monitor (middle right) - Running processes
    9. ECU Console (bottom) - Live log output

    1. Verify system health:

    In ECU Hardware Status: - All ECUs should show "Running" state - CPU and Memory columns display values - No error indicators present

    In ECU Console: - Normal startup messages appear - No error messages in red

    Additional monitoring panels show detailed information:

    Part 3: CAN Log Analysis¶

    Step 9: Switch to CAN Log Analysis Mode¶

    Prepare for data analysis and playback.

    1. Switch modes:
    2. Click View → CAN Log Analysis from the menu

    3. The layout reconfigures for signal analysis

    4. Understand the new layout:

    5. Signal Viewer (left) - Hierarchical signal tree
    6. Network Traffic (center) - Live CAN frames
    7. Signal Chart (right) - Time-series plots
    8. Signal Player & Injection (bottom) - Playback controls

    Step 10: Load and Configure Playback Data¶

    Load your captured BLF log file for analysis.

    1. Open the Signal Player panel:
    2. Locate it at the bottom of the screen

    3. If not visible, use View → Panels → Signal Player

    4. Load the BLF file:

    5. Click Select BLF button
    6. Navigate to your log file: rpm_fluctuation_capture.blf
    7. Click Open

    8. The file loads and displays statistics

    9. Inspect the loaded data:

    10. Click Inspect bundle to see details:
      • Total frames in the log
      • Duration of the capture
      • Detected CAN channels
      • Frame rate statistics

    1. Verify DBC mapping:
    2. The Networks list shows your Powertrain-CAN
    3. Confirm the DBC file is properly associated
    4. Check that signal decoding is active

    Step 11: Start Playback and Monitor Signals¶

    Begin replaying the captured data through your system.

    1. Start playback:
    2. Ensure the system is running (toolbar shows Stop button)
    3. In Signal Player, click the Play button
    4. Playback begins at 1x speed

    5. The progress bar shows current position

    6. Monitor live traffic:

    7. Switch to the Network Traffic panel
    8. CAN frames appear as they're replayed:
      • Timestamp updates continuously
      • CAN IDs show in hexadecimal
      • Data bytes display in the data column
      • Frame count increments

    1. Use traffic filters (if needed):
    2. To focus on specific messages:
      • Enter a CAN ID in the filter field (e.g., 0x100)
      • Or filter by data pattern
      • Click Apply Filter
    3. To see all traffic again:
      • Clear the filter field
      • Click Clear Filter

    Step 12: Analyze Decoded Signals¶

    Examine the decoded signal values to identify issues.

    1. Open Signal Viewer:
    2. Click on the Signal Viewer panel (left side)

    3. The tree shows all signals from your DBC

    4. Find relevant signals:

    5. Use the search box at the top
    6. Search for "RPM" to find engine speed
    7. The tree filters to matching signals

    8. Clear search to see all signals again

    9. Examine signal details:

    10. Expand CAN frames to see their signals:
      • Click the arrow next to frame 0x100 (EngineData)
      • Signals appear indented below
      • Live values update during playback
    11. Key signals to monitor:
      • Engine_RPM - Current engine speed
      • Throttle_Position - Accelerator pedal position
      • Vehicle_Speed - Current vehicle speed

    1. Select signals for charting:
    2. Click the checkbox next to Engine_RPM
    3. Select Throttle_Position
    4. Select Vehicle_Speed
    5. These signals are now ready for plotting

    Step 13: Chart and Analyze Signal Patterns¶

    Visualize signal behavior over time to identify anomalies.

    1. View the Signal Chart:
    2. Click on the Signal Chart panel (right side)
    3. Selected signals appear as colored lines
    4. Time axis shows along the bottom

    5. Value axis shows on the left

    6. Configure chart settings:

    7. Time Window: Set to 30 seconds for detail
    8. Auto-scale: Enable to fit all values
    9. Grid: Enable for easier reading

    10. Legend: Shows signal names and colors

    11. Identify the anomaly:

    12. Play through the log while watching the chart
    13. Look for unusual patterns:
      • RPM spikes without throttle input
      • Delayed throttle response
      • Oscillations in steady-state
    14. When you spot an issue:
      • Click Pause in Signal Player
      • Note the timestamp

    1. Zoom into the problem area:
    2. Use mouse wheel to zoom time axis
    3. Click and drag to pan along timeline
    4. Adjust Time Window to 5 seconds for detail

    5. The anomaly should be clearly visible

    6. Document findings:

    7. Take a screenshot of the chart:
      • Right-click on chart
      • Select Export Chart
      • Save as rpm_anomaly_chart.png
    8. Note the exact timestamp range
    9. Record signal values at the anomaly

    Part 4: Signal Injection Testing¶

    Step 14: Prepare for Signal Injection¶

    Test how the system responds to controlled inputs.

    1. Understanding injection capabilities:
    2. Signal injection sends CAN frames to the network
    3. Useful for testing ECU responses
    4. Can replay specific signal patterns

    5. Helps validate fixes and theories

    6. Access injection controls:

    7. In the Signal Player & Injection panel
    8. Look for the Injection section
    9. Controls include:
      • Start all - Begin injection
      • Pause all - Temporarily stop
      • Stop all - End injection
      • Signal configuration options

    1. Safety considerations:

    WARNING: Signal injection can affect running ECUs. - Only inject into test bench systems - Never inject into production vehicles - Verify ECU configuration allows test inputs - Monitor system response carefully

    Step 15: Configure and Execute Injection¶

    Set up a controlled test to reproduce the issue.

    1. Select signals for injection:
    2. In Signal Viewer, identify the signals to inject

    3. Common injection targets:

      • Throttle position sensor values
      • Speed sensor inputs
      • Temperature readings

    4. Configure injection parameters:

    5. Signal values to inject
    6. Injection rate (messages per second)
    7. Duration of injection

    8. Pattern (constant, ramp, sine wave)

    9. Start injection:

    10. Ensure system is running
    11. Clear Network Traffic to see new data
    12. Click Start all in injection controls

    13. Monitor the response:

      • Network Traffic shows injected frames
      • Signal Chart displays the injected values
      • ECU Console may show diagnostic messages

    14. Analyze system response:

    15. Does the RPM fluctuation reproduce?
    16. How do other signals respond?
    17. Any error codes generated?

    18. System stability maintained?

    19. Stop injection:

    20. Click Stop all when test completes
    21. Allow system to return to idle state
    22. Document the results

    Part 5: Data Export and Documentation¶

    Step 16: Export Analysis Data¶

    Save your findings for reporting and further analysis.

    1. Export network traffic:
    2. In Network Traffic panel
    3. Click Export button
    4. Choose format:
      • CSV for spreadsheet analysis
      • BLF for replay in other tools
    5. Name file: analysis_traffic_export.csv

    6. Click Save

    7. Export signal data:

    8. Ensure relevant signals are selected
    9. Time range includes the anomaly
    10. Export includes decoded values

    11. Save as: signal_analysis_export.csv

    12. Export chart images:

    13. Right-click on Signal Chart
    14. Select Export Chart
    15. Save as PNG or PDF
    16. Name: rpm_anomaly_chart.png

    Step 17: Create Analysis Summary¶

    Document your findings for the team.

    1. Compile findings:
    2. Anomaly timestamp: Exact time of issue
    3. Affected signals: List all involved signals
    4. Pattern description: What happened
    5. Injection results: How system responded

    6. Potential causes: Your analysis

    7. Prepare deliverables - organize your files in this structure:

    8. PowertrainAnalysis/ - Main project directory

      • PowertrainAnalysis.json - System configuration file
      • exports/ - Export directory
      • analysis_traffic_export.csv - Network traffic data
      • signal_analysis_export.csv - Signal analysis data
      • rpm_anomaly_chart.png - Chart screenshot
      • logs/ - Log files directory
      • rpm_fluctuation_capture.blf - Original capture file
      • analysis_summary.txt - Analysis summary document

    9. Write summary document with the following sections:

    Document Header:

    • Title: "Powertrain Analysis Summary"
    • Date: [Current Date]
    • Engineer: [Your Name]

    Issue Description:

    • Primary issue: RPM fluctuations during steady-state cruise
    • Symptoms observed during testing

    Key Findings:

    • Anomaly timestamp: 14:32:15.234
    • RPM spike details: 2000 to 2500 RPM without throttle input
    • Pattern frequency: Repeats every 30-45 seconds
    • Correlation: Linked to transmission shift points

    Injection Testing Results:

    • Test performed: Throttle position injection
    • Result: Successfully reproduced the issue
    • Observation: System response delayed by 200ms
    • Conclusion: Suggests control loop timing issue

    Recommendations:

    • Review transmission ECU shift logic
    • Check CAN bus utilization during shifts
    • Verify control loop timing parameters
    • Consider adjusting control loop response time

    Part 6: System Cleanup¶

    Step 18: Stop and Clean Up¶

    Properly shut down the system after analysis.

    1. Stop the running system:
    2. Click the Stop button in toolbar
    3. Confirm when prompted
    4. Wait for "System stopped" message

    5. All containers terminate gracefully

    6. Save final system state:

    7. File → Save to preserve any changes
    8. System configuration is updated

    9. Analysis setup is preserved

    10. Clean up resources (if needed):

    11. In terminal: docker ps to check containers
    12. docker system prune to free space
    13. Remove temporary export files if not needed

    Troubleshooting Guide¶

    Common Issues and Solutions¶

    System Won't Start¶

    Symptom: Start button clicked but ECUs don't run

    Solutions:

    1. Check Docker Desktop is running
    2. Verify no port conflicts (8080, 8081)
    3. Review ECU Console for error messages
    4. Try Mock Mode if Docker issues persist

    No CAN Traffic Visible¶

    Symptom: Network Traffic panel stays empty

    Solutions:

    1. Verify system is running (not just built)
    2. Check BLF file is loaded in Signal Player
    3. Press Play in Signal Player (not just load)
    4. Confirm DBC file is properly associated
    5. Check Network Traffic filters aren't too restrictive

    Signals Not Decoding¶

    Symptom: See CAN IDs but no signal names

    Solutions:

    1. Verify DBC file path in network properties
    2. Check DBC file contains definitions for your CAN IDs
    3. Reload the system after DBC changes
    4. Ensure Signal Viewer panel is refreshed

    Playback Not Working¶

    Symptom: Play button disabled or nothing happens

    Solutions:

    1. System must be running first
    2. BLF file must be loaded
    3. Check file isn't corrupted
    4. Verify sufficient memory for large logs
    5. Try smaller BLF file to test

    Chart Not Updating¶

    Symptom: Signal Chart remains empty

    Solutions:

    1. Select signals in Signal Viewer first
    2. Ensure playback is running
    3. Check time window settings
    4. Verify Auto-scale is enabled
    5. Refresh the chart panel

    Injection Not Sending¶

    Symptom: Start injection but no frames appear

    Solutions:

    1. System must be running
    2. Network must be properly configured
    3. Check injection isn't paused
    4. Verify signal configuration
    5. Monitor ECU Console for errors

    Tips for Effective Analysis¶

    Performance Optimization¶

    1. For large BLF files:
    2. Close unnecessary panels to free memory
    3. Use filters to reduce displayed data
    4. Export smaller time ranges

    5. Process in segments if needed

    6. For smooth playback:

    7. Limit number of charted signals
    8. Reduce Signal Chart time window
    9. Disable auto-scroll in Network Traffic
    10. Close other applications

    Analysis Best Practices¶

    1. Signal selection:
    2. Start with key signals only
    3. Add related signals as needed
    4. Group similar signals for comparison

    5. Use consistent time windows

    6. Documentation:

    7. Screenshot important findings immediately
    8. Note exact timestamps
    9. Export data before making changes

    10. Keep analysis notes in text file

    11. Injection testing:

    12. Start with known-good values
    13. Change one parameter at a time
    14. Monitor all related signals
    15. Stop immediately if instability occurs

    Advanced Topics¶

    Working with Multiple Networks¶

    If your system has multiple CAN buses:

    1. Add additional CAN networks to canvas
    2. Connect ECUs to appropriate buses
    3. Load separate DBC for each network
    4. Switch between networks in Network Traffic
    5. Filter signals by network in Signal Viewer

    Custom ECU Applications¶

    For ECUs running custom applications:

    1. Ensure application uses SocketCAN
    2. Configure CAN interface in ECU
    3. Build application into container
    4. Monitor application logs in ECU Console
    5. Use Terminal panel for debugging

    Integration with External Tools¶

    Export data for analysis in other tools:

    1. MATLAB/Simulink:
    2. Export as CSV with timestamps
    3. Import into MATLAB workspace

    4. Process with Signal Processing Toolbox

    5. Python Analysis:

    6. Export as CSV
    7. Use pandas for data manipulation
    8. Plot with matplotlib

    9. Perform statistical analysis

    10. CANalyzer/CANoe:

    11. Export as BLF
    12. Import into Vector tools
    13. Apply existing test configurations
