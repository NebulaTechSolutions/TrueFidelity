    End-to-End Example¶

    This tutorial walks through a complete TrueFidelity workflow: project creation, playback analysis, signal inspection, data injection, annotation, and reporting. Expect to spend 20–30 minutes.

    Scenario¶

    You’re investigating intermittent braking anomalies captured during a proving-ground test. You have: - Two CAN log files (baseline and suspect run) - A DBC file with signal definitions - A test plan requiring annotations, injection replay, and a formal report

    Step 1 – Prepare the Project¶

    1. Create a new project named Brake-Analysis in your project workspace.
    2. Import the baseline log (baseline.mf4) and the suspect log (suspect.mf4).
    3. Import the braking.dbc definition file.
    4. Save the project.

    Step 2 – Configure Layout¶

    1. Load the Analysis layout (or create one):
    2. Timeline top
    3. Messages left
    4. Charts right
    5. Signals docked below Messages
    6. Annotations bottom
    7. Pin key signals: brake_pedal, wheel_speed_fl, wheel_speed_fr, hydraulic_pressure.
    8. Drag the pinned signals into Charts.

    Step 3 – Compare Baseline vs. Suspect¶

    1. Enable both logs; color-coding distinguishes them.
    2. Align them by timestamp. Use the Align logs feature with the start of the test run.
    3. Play both logs at 1× speed.
    4. Watch for diverging wheel speeds or pressure spikes.
    5. Pause when you spot a discrepancy (e.g., hydraulic pressure spike without pedal input).

    Step 4 – Annotate Events¶

    1. Add an annotation named “Pressure spike without pedal” at the anomaly.
    2. Tag with braking, investigate, suspect.
    3. Capture a chart screenshot for later.
    4. Add a bookmark slightly earlier labeled “Baseline reference” for comparison.

    Step 5 – Filter for Related Signals¶

    1. Apply a message filter for IDs associated with the braking ECU.
    2. Enable Filter to Range around the anomaly (10 seconds window).
    3. Inspect payload differences between baseline and suspect logs.
    4. Add a second annotation if you notice a diagnostic code or unusual payload bit set.

    Step 6 – Injection Reproduction (Bench Setup)¶

    1. Switch to the Injection layout (Data Injection panel left, Messages center, Charts right).
    2. Create a message template replicating the suspect payload.
    3. Configure a burst schedule: send 20 frames at 20 Hz.
    4. Start injection on the bench ECU (ensure it’s disconnected from the vehicle).
    5. Monitor the ECU response. Chart hydraulic_pressure and brake_warning_flag signals (from live feed or recorded response).
    6. Annotate the start/end of the injection and note whether the anomaly reproduces.

    Step 7 – Export Data¶

    1. Select the anomaly range on the timeline.
    2. Export to CSV with decoded signals and annotations included.
    3. Export annotation CSV for the entire session.
    4. Save injection profile (pressure_spike_replay.tfinject) for future use.

    Step 8 – Build the Report¶

    1. Open Report Builder.
    2. Choose the Validation template.
    3. Fill in overview information: test objective, logs used, hardware setup.
    4. Add the annotations created earlier.
    5. Insert the chart screenshot showing baseline vs. suspect difference.
    6. Attach the CSV export and injection profile as supporting files.
    7. Generate the PDF report and review for accuracy.

    Step 9 – Archive & Share¶

    1. Save the project.
    2. Zip the project folder including exports, profiles, and report.
    3. Upload to your collaboration platform or attach to the investigation ticket.
    4. Document next steps (e.g., replicate on hardware-in-the-loop rig).

    Reflection¶

    • What triggered the anomaly? Document hypotheses in the project notes.
    • Did the injection reproduce the issue? If not, what differed between bench and road conditions?
    • Are additional logs needed? Plan follow-up captures if necessary.

    Variations¶

    • Swap logs to evaluate other subsystems (steering, acceleration).
    • Extend the workflow with automated CLI exports.
    • Integrate hardware signals from auxiliary bus captures (LIN, FlexRay) if your build supports it.
