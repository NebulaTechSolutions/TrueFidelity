    Injection Scenario Walkthrough¶

    This tutorial demonstrates how to design, execute, and analyze a controlled data injection scenario. Estimated time: 25 minutes.

    Scenario¶

    A suspension controller occasionally fails to set a diagnostic flag when subjected to a specific message pattern. Engineers suspect timing jitter on a CAN frame is responsible. Goal: recreate the pattern using injection, confirm the fault, and document results.

    Prerequisites¶

    • Bench setup with the suspension controller connected to can0
    • TrueFidelity with injection feature enabled
    • DBC file suspension.dbc
    • Baseline log capturing normal behavior

    Step 1 – Project Setup¶

    1. Create a project Suspension-Injection-Test.
    2. Import baseline_suspension.mf4 and suspension.dbc.
    3. Load the Injection layout (Data Injection panel left, Messages center, Charts right).

    Step 2 – Analyze Baseline¶

    1. Play the baseline log to identify normal timing.
    2. Pin signals: suspension_status, damping_pressure, diagnostic_flag.
    3. Add them to Charts and capture reference screenshots.
    4. Note message ID and interval for the suspect frame (e.g., ID 0x350, every 20 ms).

    Step 3 – Create Injection Template¶

    1. In Data Injection panel click Add Message.
    2. ID: 0x350, DLC: 8, Payload: AA 00 3C 10 00 00 00 01.
    3. Set channel to can0.
    4. Add a second message template representing the jitter (same ID, different payload bit).

    Step 4 – Configure Schedules¶

    • Template A (normal message): Interval 20 ms.
    • Template B (jitter): Burst of 3 frames at 5 ms intervals every 200 ms.
    • Ensure Template B runs slightly offset (delay 50 ms) to overlap with Template A.

    Tip

    Use the preview graph in the injection panel to verify the combined effect of both schedules before starting.

    Step 5 – Safety Checks¶

    1. Confirm the bench controller is in test mode.
    2. Ensure an emergency stop or power cutoff is within reach.
    3. Announce to teammates that injection is starting.

    Step 6 – Run Injection¶

    1. Press Start in the injection panel.
    2. Monitor counters to ensure messages send successfully.
    3. Watch Messages panel filtered for ID 0x350 to verify interval changes.
    4. Observe Charts for diagnostic_flag response.
    5. Let injection run for at least 2 minutes.

    Step 7 – Annotate Findings¶

    • When the diagnostic flag fails to assert, add an annotation “Diagnostic miss #1”.
    • Note conditions: time, temperature (if available), injection profile version.
    • Add bookmarks for times when the flag works correctly for comparison.

    Step 8 – Adjust and Retest¶

    1. Stop injection.
    2. Modify Template B burst rate (e.g., increase to 5 frames) and restart.
    3. Record whether the failure becomes more frequent.
    4. Repeat until you establish a reliable reproduction pattern.

    Step 9 – Export Evidence¶

    1. Export a segment covering a successful reproduction (include annotations).
    2. Export the injection profile (suspension_jitter_profile.tfinject).
    3. Capture chart screenshots showing command vs. diagnostic flag behavior.
    4. Download the injection log from the panel for auditing.

    Step 10 – Build the Report¶

    1. Launch Report Builder with the Injection Validation template.
    2. Document setup (hardware, firmware versions, environment conditions).
    3. Insert annotations and screenshots.
    4. Attach exported segment, injection profile, and logs.
    5. Summarize findings and recommended next steps (e.g., adjust controller debounce logic).

    Cleanup¶

    • Stop injection and power down the controller if required.
    • Restore the system to normal operating mode.
    • Commit project changes to version control.

    Lessons Learned¶

    • Controlled injection can reproduce rare timing issues efficiently.
    • Logging annotations and exporting profiles create a repeatable test package.
    • Combining burst and interval schedules allows complex message patterns without scripting.

    Next Steps¶

    • Share the report with the suspension firmware team.
    • Use the same profile on hardware-in-the-loop rigs for broader validation.
    • Explore Safety & Best Practices to formalize injection protocols.
